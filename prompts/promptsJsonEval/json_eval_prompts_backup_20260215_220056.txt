**Block B: Current Task (Reference vs. Candidate)**

You will now be presented with two JSON objects: `[[Json_Reference]]` (the Ground Truth, or human-validated extraction) and `[[Json_Candidate]]` (the LLM-generated extraction). Your task is to compare the `[[Json_Candidate]]` against the `[[Json_Reference]]` field by field, using the following guidelines:

**Evaluation Criteria:**

*   **Factual Accuracy (Quantitative):** For numerical values and units (e.g., size, quantum yield, elemental composition), the `[[Json_Candidate]]` MUST match the `[[Json_Reference]]` almost exactly. Scientifically negligible differences are acceptable (e.g., rounding errors).
    *   *Strict Rule:* If the `[[Json_Reference]]` contains a specific number (e.g., "50%"), and the `[[Json_Candidate]]` contains a significantly different number (e.g., "90%"), this is a Critical Error and the Candidate is "Bad".

*   **Descriptive Accuracy (Qualitative - FLEXIBLE):**
    *   **Information Gain/Enrichment:** If the `[[Json_Candidate]]` includes valid chemical adjectives or functional descriptions NOT present in the `[[Json_Reference]]` (e.g., `[[Json_Reference]]`: "Carbon Dots"; `[[Json_Candidate]]`: "Fluorescent Carbon Dots with peroxidase activity"), and this information is scientifically plausible and does not contradict the `[[Json_Reference]]`, treat this as CORRECT/ENRICHMENT and mark it as "Good".

This is a Critical Error and the Candidate is "Bad".
    *   **DO NOT PENALIZE:** If the `[[Json_Candidate]]` fills a `null` field in the `[[Json_Reference]]` with scientifically plausible data. This is considered a Discovery and should not be penalized.

*   **Omission Check:** If the `[[Json_Candidate]]` has `null` or an empty string where the `[[Json_Reference]]` has specific data, this indicates a failure to extract information. This is an Error and the Candidate is "Bad".
